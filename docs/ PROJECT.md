# Project Guide

## Описание проекта

Это frontend-проект для мобильного веб-сценария сканирования лица через браузер телефона.

Пользователь проходит несколько шагов:

1. Страница с описанием проекта.
2. Страница ввода данных пользователя.
3. Страница с инструкцией перед сканированием.
4. Страница сканирования через камеру телефона.
5. Отправка данных сканирования на backend.
6. Страница с результатами.

Проект должен быть простым, понятным, адаптированным под мобильные устройства и готовым к дальнейшему развитию.

## Главная цель

Создать аккуратный frontend-проект без лишнего кода, сложных абстракций и хаотичной структуры.

Архитектура должна быть простой, но такой, чтобы проект можно было развивать по версиям.

## Основной пользовательский сценарий

Пользователь открывает сайт.

Сначала он видит страницу с описанием сервиса.

После этого пользователь вводит свои данные.

Затем пользователь видит инструкцию по правильному сканированию.

После инструкции открывается камера телефона через браузер.

SDK проверяет лицензию.

SDK определяет лицо пользователя.

Если лицо валидно, SDK запускает процесс сканирования.

Frontend получает данные и показатели от SDK.

После завершения сканирования frontend отправляет данные на backend.

Backend возвращает результаты.

Frontend отображает результаты пользователю.

## Важные части проекта

В проекте будут:

- мобильный UI/UX;
- страницы сценария;
- форма ввода данных пользователя;
- инструкция перед сканированием;
- работа с камерой телефона;
- интеграция с SDK;
- проверка лицензии через SDK;
- определение и валидация лица через SDK;
- получение показателей от SDK;
- отправка данных на backend;
- отображение результатов от backend.

## Правила архитектуры

- Не усложнять проект без необходимости.
- Не создавать лишние папки и файлы.
- Не писать код “на будущее”.
- Один файл должен отвечать за одну понятную задачу.
- Страницы должны быть простыми и понятными.
- UI-компоненты не должны напрямую работать с SDK.
- UI-компоненты не должны напрямую содержать сложную backend-логику.
- SDK-логика должна быть изолирована отдельно.
- API-запросы к backend должны быть изолированы отдельно.
- Конфигурация, например backend URL, должна храниться отдельно.
- Проект должен быть удобен для дальнейшего развития.

## Предполагаемые страницы

- Welcome page — описание проекта.
- User data page — ввод данных пользователя.
- Scan instruction page — инструкция перед сканированием.
- Scan page — камера, SDK и процесс сканирования.
- Results page — отображение результатов.

## Backend

Frontend будет отправлять данные на backend.

Backend URL должен быть вынесен в конфигурацию или env-переменные.

Нельзя хардкодить backend URL прямо в компонентах.

Все запросы к backend должны быть вынесены в отдельный слой, чтобы их было легко менять и поддерживать.

## SDK

SDK используется для:

- проверки лицензии;
- открытия камеры;
- определения лица;
- проверки валидности лица;
- запуска сканирования;
- получения показателей.

SDK нельзя вызывать напрямую из UI-компонентов.

SDK-логику нужно изолировать в отдельные файлы, чтобы страницы и компоненты оставались простыми.

## UI/UX

Проект в первую очередь должен быть удобным на мобильных устройствах.

Интерфейс должен быть:

- простой;
- понятный;
- аккуратный;
- адаптивный;
- без перегруженных экранов.

Каждый шаг должен вести пользователя дальше по сценарию.

## Правила для Cursor

Перед созданием архитектуры или изменением кода нужно прочитать этот файл.

Cursor должен:

- предложить простую структуру проекта;
- не создавать слишком сложную архитектуру;
- не добавлять лишние библиотеки;
- не создавать много файлов без необходимости;
- разделить страницы, компоненты, SDK-логику и API-логику;
- сделать структуру понятной для дальнейшего развития;
- учитывать, что проект будет развиваться по версиям;
- сохранять код простым и читаемым.

## Что важно

Сейчас не нужно делать финальную сложную архитектуру.

Нужно создать понятный стартовый каркас проекта, который можно будет постепенно развивать.
